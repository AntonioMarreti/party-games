# Party Battle Content Strategy

Этот файл фиксирует практичную стратегию наполнения Party Battle контентом.

Главная мысль:

- нам нужен не просто "смешной текст";
- нам нужны `playable cards`, которые хорошо работают внутри конкретного режима;
- поэтому у каждого режима должен быть свой pipeline, а не один общий blind import.

## Базовый принцип

Нужно разделять:

1. сырьё
2. фильтрацию
3. трансформацию в игровой формат
4. ручную проверку
5. canonical pack

Сырьё почти никогда не равно готовой карточке для Party Battle.

## По режимам

### `bluff`

Что нужно режиму:

- короткий понятный факт;
- одна ясная "правда";
- без длинной энциклопедичности;
- без слишком нишевого знания;
- без тяжёлой медицины, криминала, политики и мрака.

Лучшие источники:

- trivia/fact datasets
- weird facts lists
- короткие энциклопедические факты

Подход:

- importer-first
- потом лёгкая ручная чистка

Pipeline:

1. импорт фактов
2. фильтр на длину и понятность
3. выбросить скучные, слишком академические и токсичные
4. закоммитить в `bluff/*`

Статус:

- это самый перспективный режим для автоматизированного импорта

### `advice`

Что нужно режиму:

- бытовая, неловкая, знакомая ситуация;
- короткая формулировка;
- сразу понятно, как игроку отвечать вредным советом;
- вопрос должен быть social/playful, а не informational.

Лучшие источники:

- question-like corpora
- бытовые Q&A выгрузки
- житейские awkward questions
- `ru_qna_333k` как сырьё

Подход:

- importer + жёсткий curation

Pipeline:

1. брать только вопросные формулировки
2. выбрасывать:
   - медицину
   - право
   - политику
   - школу/экзамены
   - техподдержку
   - прямую жесть
3. приводить к playable lead’ам:
   - `Что делать, если...`
   - `Как выкрутиться, если...`
   - `Как объяснить, что...`
4. вручную дочищать слабые или слишком серьёзные

Статус:

- importer полезен
- но без post-filter review качество быстро разваливается

### `whoami`

Что нужно режиму:

- вопрос, по которому весело голосовать за игроков;
- social/personality framing;
- вопрос должен провоцировать сравнение людей в комнате;
- минимум абстракции и максимум узнаваемости.

Лучшие источники:

- party questions
- friendship questions
- office/social prompts
- custom curated lists

Подход:

- importer/manual hybrid

Pipeline:

1. собрать social-question source
2. выбросить всё, что требует внешних знаний
3. оставить только то, что работает в группе друзей/коллег
4. разложить по темам: `office`, `party`, `friendship`, `provocative`

Статус:

- режим хорошо живёт от curated thematic packs

### `joke`

Что нужно режиму:

- короткий setup;
- недосказанность;
- понятная точка, куда игрок может "добить";
- не готовый анекдот;
- не dated-reference ради dated-reference.

Плохие источники:

- старые joke datasets
- готовые анекдоты
- длинные истории с punchline

Лучшие источники:

- AI-assisted generation
- вручную написанные setup-пакеты
- curated comedy premises

Подход:

- generation-first, не importer-first

Pipeline:

1. генерировать короткие setup’ы по шаблонам
2. ручной review
3. выбросить:
   - законченные анекдоты
   - политоту
   - dated references
   - токсичную грязь без игрового value
4. паковать в `joke/*`

Хорошая форма:

- `Если на первом свидании всё пошло слишком неловко, то...`
- `Самое странное, что можно услышать на семейном ужине...`

Плохая форма:

- готовая шутка, где уже некуда отвечать

Статус:

- blind import сюда почти не помогает

### `meme`

Что нужно режиму:

- короткий reaction-prompt;
- шаблон, который хорошо маппится на GIF;
- высокая читаемость с первого взгляда;
- минимум длинных предложений.

Плохие источники:

- joke corpora
- длинные смешные истории
- всё, что уже само по себе punchline

Лучшие источники:

- curated template library
- AI generation по шаблонам
- ручной набор коротких prompts

Подход:

- generation-first

Рабочие шаблоны:

- `Когда...`
- `Ты, когда...`
- `Опять...`
- `Утро после...`
- `Когда понял, что...`
- `Твоё лицо, когда...`

Pipeline:

1. задать шаблонные формы
2. генерировать пачки коротких prompt’ов
3. выбросить всё, что:
   - слишком длинное
   - слишком нишевое
   - токсичное
   - не маппится на реакционный GIF
4. держать curated base

Статус:

- importer из joke datasets сюда почти не годится

### `caption`

Что нужно режиму:

- хороший visual;
- читаемый кадр;
- понятная эмоция/ситуация;
- картинка должна провоцировать подпись.

Источник контента:

- curated GIF/image pool
- локальный manifest ссылок
- в идеале позже локальные assets или стабильный internal pool

Подход:

- visual curation first

Pipeline:

1. собрать список надёжных visual assets
2. проверить, что они не битые и не скучные
3. проверить, что они читаются без внешнего контекста
4. положить в canonical `caption/*`

Статус:

- главный контент здесь это не тексты, а визуалы

## Что importer реально должен делать

Importer полезен не для всего подряд, а как инфраструктура для конкретных режимов.

### Где importer уместен

- `bluff`
- `advice`
- частично `whoami`

### Где importer ограниченно полезен

- `joke`

Только как источник сырья для дальнейшей ручной трансформации, но не как прямой путь в pack.

### Где importer почти не решает задачу

- `meme`
- `caption`

Там важнее curated generation и curated visual pool.

## Что не делать

- не импортировать старые анекдоты вслепую в `joke/base`
- не использовать joke datasets как основной source для `meme`
- не пытаться слепо наполнить `caption` текстовыми источниками
- не считать "большой объём" успехом, если карточки плохо играются

## Practical roadmap

### Stage 1

- добить `bluff` content pipeline
- расширить `advice` из question-like sources

### Stage 2

- собрать curated visual pool для `caption`
- решить, нужен ли отдельный `caption/18plus`

### Stage 3

- сделать generation rules для `joke`
- сделать generation rules для `meme`

### Stage 4

- подключить и использовать уже существующие thematic packs, где они реально готовы
- после этого делать manual cleanup по live-feedback

## Практический вывод

Следующий самый полезный ход:

1. не мучить blind import для `joke/meme`;
2. использовать importer там, где он реально силён: `bluff`, `advice`, `whoami`;
3. отдельно строить curated/generation pipelines для `joke`, `meme`, `caption`.
